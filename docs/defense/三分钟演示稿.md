# 三分钟系统演示稿

## 0:00 - 0:30 项目定位

各位老师好，我的毕业设计题目是“基于容器编排的微服务弹性伸缩策略研究与实践”。这个项目不是传统业务管理系统，而是一套微服务弹性伸缩实验与评估系统。它的目标是在 Kubernetes 环境中比较不同伸缩策略在突发负载下的性能、资源利用率和稳定性。

## 0:30 - 1:10 系统结构

系统主要分为五部分：

1. `services/sample-api` 是实验用微服务，提供业务接口和 Prometheus 指标。
2. `deploy/base` 是基础 Kubernetes 部署，包括 Deployment 和 Service。
3. `deploy/hpa`、`deploy/vpa` 是伸缩策略配置。
4. `deploy/monitoring` 是 Prometheus 和 Prometheus Adapter，用来采集并暴露 QPS 自定义指标。
5. `loadtest`、`scripts`、`results` 负责压测、自动化执行和结果分析。

## 1:10 - 1:55 实验流程

实验流程是：先启动 Minikube 并部署 `sample-api`，然后安装 metrics-server、Prometheus、Prometheus Adapter 和 VPA。接着使用 Locust 生成突发负载，分别运行 `static`、`hpa_cpu`、`hpa_multi`、`hpa_vpa` 四组策略。每次运行都会保存 Locust CSV、HPA/VPA 状态、Kubernetes events、Pod 资源采样和 manifest。最后通过 Python 脚本汇总数据并生成图表。

## 1:55 - 2:35 策略对比

四组策略的含义是：

- `static` 是固定 2 个副本，作为基线。
- `hpa_cpu` 只看 CPU 利用率。
- `hpa_multi` 同时看 CPU、Memory 和 QPS。
- `hpa_vpa` 在多指标 HPA 的基础上加入 VPA Initial，让 Pod 创建时使用更合适的资源请求。

## 2:35 - 3:00 结果结论

正式实验在 `burst` 场景下每组运行 3 次，共 12 条有效记录。结果显示，`hpa_vpa` 组整体 P95 延迟最低、QPS 最高、失败率最低；固定副本策略在突发负载下失败率和成本代理指标较高。我的结论是：在微服务突发负载场景中，引入业务指标 QPS 和资源建议机制，可以比单纯 CPU-HPA 更好地平衡性能和资源利用率。
