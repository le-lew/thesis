# 答辩问答速查

## 1. 你这个系统到底是什么？

这是一个微服务弹性伸缩实验与评估系统。它把示例微服务部署到 Kubernetes 中，通过 Locust 制造突发负载，再用 HPA、VPA、Prometheus 等组件采集指标和执行伸缩，最后汇总数据生成论文图表。

## 2. 为什么选 Kubernetes？

Kubernetes 是云原生领域主流的容器编排平台，内置 Deployment、Service、HPA 等机制，也能通过 Prometheus Adapter 接入自定义指标，适合研究微服务弹性伸缩策略。

## 3. HPA 和 VPA 有什么区别？

HPA 是水平伸缩，主要调整 Pod 副本数；VPA 是垂直伸缩，主要给单个 Pod 推荐或调整 CPU/Memory 请求值。本项目中 HPA 负责应对流量变化，VPA 使用 Initial 模式，在 Pod 创建时应用资源建议，减少运行中频繁重建带来的干扰。

## 4. 为什么只看 CPU 不够？

CPU 只能反映资源压力，不能完整反映业务流量变化。有些场景中请求数上升早于 CPU 明显升高，如果只看 CPU，扩容可能滞后。因此本项目加入 QPS 作为业务指标，并与 CPU、Memory 一起构成多指标 HPA。

## 5. QPS 怎么进入 HPA？

`sample-api` 暴露 Prometheus 格式的 `http_requests_total` 指标，Prometheus 采集后，Prometheus Adapter 使用 `rate(http_requests_total[1m])` 转换为 `http_requests_per_second`，再注册到 Kubernetes 的 `custom.metrics.k8s.io`，HPA 就可以把它作为 Pods 类型指标使用。

## 6. 为什么 VPA 用 Initial？

VPA 的 Auto 模式可能在运行中驱逐并重建 Pod，会影响压测稳定性。Initial 模式只在 Pod 创建时应用资源建议，更适合作为 HPA 协同实验中的稳妥方案，避免水平伸缩和垂直伸缩同时频繁调整造成干扰。

## 7. 实验为什么要重复 3 次？

Kubernetes 调度、Pod 启动、负载生成和本机资源状态都会带来波动。重复 3 次可以减少偶然因素影响，并用均值和标准差展示趋势，让结论更可信。

## 8. 成本代理指标怎么算？

成本代理指标基于 CPU 和内存使用量加权估算，再换算到每 1000 次请求。它不是云厂商真实账单，而是为了在相同实验条件下横向比较不同策略的资源开销。

## 9. 为什么结果不能保证跨机器完全一致？

因为实验运行在本地 Minikube，机器 CPU、内存、Docker 状态、Kubernetes 调度和镜像缓存都会影响瞬时结果。本项目强调的是流程可重跑、数据可追溯、趋势可复现，而不是跨机器数值逐点一致。

## 10. 你的主要结论是什么？

在正式 `burst` 场景下，`hpa_vpa` 组 P95 延迟最低、QPS 最高、失败率最低；固定副本策略在突发流量下更容易出现失败和较高成本代理值；多指标 HPA 比单 CPU HPA 更能反映真实业务压力。
